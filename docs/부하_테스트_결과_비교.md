## 2. 전체 최적화 타임라인

### Phase 1: 1차 최적화 (2026-03-11 ~ 03-18)
- Quiz: `ORDER BY RANDOM()` → UUID 메모리 캐싱 (O(1) 랜덤 선택)
- Quiz: `GetQuestionById` O(n) 선형 탐색 → map O(1)
- Community: `GetRanking` 60초 인메모리 캐시
- Community: `DeletePost` S3 삭제 비동기 처리
- DB 커넥션 풀 조정 (과다 100 → 적정 50)

### Phase 2: Community GetFeed 캐시 (2026-04-02)
- `GetFeed` 페이지별 인메모리 캐시 (30초 TTL)
- `COUNT(*) OVER()` window function 제거 → 별도 COUNT + 60초 캐시
- DB 복합 인덱스 `(is_admin_post DESC, created_at DESC)`
- Redis pool 10 → 30, MinIdle 5, timeout 2초
- userclient 프로필 gRPC 호출 인메모리 캐시
- UploadMedia 독립 엔드포인트 제거 (PR #191)

### Phase 3: singleflight + DB 튜닝 (2026-04-02)
- GetFeed/Count에 `singleflight` 적용 (thundering herd 방지)
- 독립 context로 연쇄 실패 방지
- DB MaxOpenConns 30 → 50, MaxIdleConns 10 → 25
- ConnMaxLifetime 5분 → 10분
- Feed 캐시 TTL 30초 → 60초, Count 캐시 TTL 60초 → 120초
- 프로필 캐시 TTL 2분 → 5분

### Phase 4: Stale-While-Revalidate + HPA (2026-04-02)
- GetFeed stale-while-revalidate: 만료 캐시 즉시 반환 + 백그라운드 갱신
- 50초마다 페이지 1~5 백그라운드 캐시 워밍
- Community HPA minReplicas 1 → 3, maxReplicas 6

## 7. 적용된 최적화 상세

### 변경 파일

| 파일 | 변경 내용 |
|------|----------|
| `community/internal/handler/handler.go` | Redis pool 30, feedCache, singleflight, 백그라운드 캐시 워밍 (50초) |
| `community/internal/handler/post.go` | GetFeed stale-while-revalidate, COUNT 분리, singleflight, refreshFeedCache/Count |
| `community/internal/userclient/client.go` | sync.Map 프로필 5분 캐시 |
| `community/main.go` | DB MaxOpenConns 50, MaxIdleConns 25, ConnMaxLifetime 10분 |
| `community/proto/community.proto` | UploadMedia 제거 (PR #191) |
| DB | `idx_posts_feed_order (is_admin_post DESC, created_at DESC)` 인덱스 |
| ArgoCD | community HPA minReplicas=3, maxReplicas=6 |
| `stress-test.js` | Quiz 429 정상 처리, GetFeed 100% |

### 핵심 아키텍처 패턴

```
요청 → 캐시 확인 → 히트: 즉시 반환 (~46ms)
                  → 만료: stale 데이터 즉시 반환 + 백그라운드 갱신
                  → 미스: singleflight로 DB 1회 조회 (최초 요청만)

백그라운드 워밍 (50초마다):
  → 페이지 1~5 미리 갱신
  → Count 캐시 미리 갱신
  → 캐시 TTL(60초) 전에 갱신되어 만료 거의 없음
```
