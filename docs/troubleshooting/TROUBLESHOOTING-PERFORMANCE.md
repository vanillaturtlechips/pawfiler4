**작성일**: 2026년 3월 11일  
**문제**: Quiz 서비스 부하 테스트에서 심각한 성능 저하 발생  
**결과**: 읽기 성능 97% 개선 완료, 쓰기 성능 추가 최적화 필요

## 📋 문제 상황

### 초기 증상
부하 테스트(10분간 150명→1,000명 동시 사용자)에서 Quiz 서비스가 극도로 느려졌습니다:

- Quiz 서비스 평균 응답시간: **3.7초** (목표: 1초 이하)
- 95% 사용자가 **10초 이상** 대기 (목표: 2초 이하)
- Community 서비스는 정상 (평균 0.4초)

### 근본 원인 분석

**1. 비효율적인 랜덤 쿼리**
```sql
-- 문제가 된 쿼리
SELECT * FROM quiz.questions 
WHERE difficulty = ? AND type = ? 
ORDER BY RANDOM() 
LIMIT 1;
```
- 매번 전체 테이블을 스캔하고 정렬
- 11개 문제밖에 없는데도 3.7초 소요
- 인덱스를 사용할 수 없는 구조

**2. 데이터베이스 연결 부족**
- Quiz 서비스: 기본 설정 (연결 수 제한)
- Community 서비스: 25개 연결 (1,000명 사용자에 부족)

**3. 캐시 미활용**
- 동일한 11개 문제를 매번 데이터베이스에서 조회
- 메모리 캐시 없이 반복적인 DB 접근

## ❌ 1차 최적화 시도 - 실패 사례

### 시도한 방법
1. **쿼리 개선**: `ORDER BY RANDOM()` → `OFFSET` 방식으로 변경
2. **연결 수 증가**: 데이터베이스 연결을 100개로 대폭 증가

### 예상과 다른 결과
- Quiz 응답시간: 3.7초 → **4.4초로 더 악화**
- Community 응답시간: 0.4초 → **1.1초로 악화**
- 전체 처리량: 42만 → **32만 요청으로 감소**

### 실패 원인
1. **OFFSET 쿼리도 여전히 느림**: COUNT(*) 실행 + N개 행 스캔 필요
2. **과도한 연결로 인한 오버헤드**: 100개 연결이 시스템 부하 증가
3. **데이터베이스 과부하**: Quiz 서비스가 DB를 과부하시켜 다른 서비스까지 영향

## ✅ 2차 최적화 - 성공 사례

### 핵심 아이디어: "데이터베이스를 아예 안 쓰자"

Quiz 문제는 자주 바뀌지 않는 정적 데이터입니다. 매번 데이터베이스에 물어볼 필요가 없다고 판단했습니다.

### 구현한 해결책

**1. 메모리 캐싱 도입**
```go
// 서비스 시작 시 모든 문제를 메모리에 로드
type QuizRepository struct {
    questions []Question  // 11개 문제 전체를 메모리에 저장
}

func (r *QuizRepository) GetRandomQuestion() Question {
    // 데이터베이스 접근 없이 메모리에서 즉시 반환
    randomIndex := rand.Intn(len(r.questions))
    return r.questions[randomIndex]
}
```

**2. 비동기 답안 처리**
```go
func (r *QuizRepository) SubmitAnswer(answer UserAnswer) error {
    // 1단계: Redis 큐에 빠르게 저장
    r.redisClient.LPush("answer_queue", answer)
    
    // 2단계: 사용자에게 즉시 성공 응답
    return nil
    
    // 3단계: 백그라운드 워커가 나중에 DB에 저장
}
```

**3. 자동 데이터 갱신**
- 30초마다 데이터베이스에서 최신 문제 목록을 다시 로드
- 새로운 문제가 추가되어도 자동으로 반영

### 놀라운 결과

**읽기 성능 (GetRandomQuestion)**
- 응답시간: 3.7초 → **0.05초** (74배 빠름)
- 성공률: **100%** (완벽)
- 데이터베이스 쿼리: **0개** (완전 제거)

**쓰기 성능 (SubmitAnswer)**
- 응답시간: 즉시 응답 (Redis 큐 활용)
- 성공률: 36% (개선 필요)
- 실제 DB 저장: 백그라운드에서 처리

**전체 시스템 (10분 테스트 결과)**
- 평균 응답시간: 2,070ms → **53ms** (97% 개선)
- 처리량: 42만 → **115만 요청** (174% 증가)
- P95 응답시간: 10,411ms → **164ms** (98% 개선)

- **캐시**: Redis (비동기 큐 + 향후 캐시 확장 가능)
- **메모리 사용량**: 55KB (11개 문제 × 5KB)
- **배포**: Kubernetes + ArgoCD

### 해결한 기술적 이슈

**진짜 문제: Redis 클라이언트 연결 부족**:
```go

redis.NewClient(&redis.Options{
    PoolSize:     20,  // ← 1000명 동시 사용자에 부족
```

### 부하 테스트 시 병목 분석 (실제 데이터)
```

→ 63.6% 연결 대기/타임아웃 실패
```

### 해결 방법

**Redis 클라이언트 최적화**:
```go
// 최적화된 설정
redis.NewClient(&redis.Options{
    Addr:         redisAddr,
    Password:     "",
    DB:           0,
    PoolSize:     100, // 20 → 100 (5배 증가)
    MinIdleConns: 20,  // 5 → 20 (4배 증가)
    DialTimeout:  5 * time.Second,  // 연결 타임아웃
    ReadTimeout:  3 * time.Second,  // 읽기 타임아웃
    WriteTimeout: 3 * time.Second,  // 쓰기 타임아웃
    MaxRetries:   3,                // 재시도 횟수
})
```

### Redis 리소스 검증

**현재 Redis 상태**:
- 메모리 사용: 1.22MB / 400MB (0.3%)
- 피크 메모리: 19.24MB (부하 테스트 시)
- 연결 수: 21개 / 10,000개

**연결 수 증가 시 예상 사용량**:

- 총 예상 사용량: 21.8MB / 400MB (5.5%)
- **결론**: 리소스 충분함 ✅

**개선 근거**:

## 🔄 HPA vs ArgoCD 충돌 문제

### 정상 작동하는 서비스 비교

**Envoy 서비스** (HPA 정상 작동):
- HPA: 10개 replicas

**Quiz 서비스** (HPA 실패):

**해결 필요**: ArgoCD 설정 수정 또는 HPA 관리 필드 예외 처리

## ✅ 4차 해결: HPA vs ArgoCD 충돌 해결

### 문제 발견

**HPA 스케일링 실패 원인**:
```yaml
# deployment.yaml
spec:
  replicas: 2  # ← 이게 문제!
```

**충돌 패턴**:
1. HPA가 CPU 186%를 감지하고 2개 → 4개로 스케일 업
2. ArgoCD selfHeal이 Git의 `replicas: 2`를 보고 되돌림
3. 무한 반복...

### 해결 방법

**Deployment에서 replicas 필드 제거**:
- HPA가 완전히 제어할 수 있도록 함
- ArgoCD는 더 이상 replicas를 관리하지 않음

참고: [ArgoCD와 HPA 충돌 해결 가이드](https://medium.com/pickme-engineering-blog/avoiding-deployment-conflicts-hpa-scaling-vs-argocd-syncing-807fb9cd2953)
