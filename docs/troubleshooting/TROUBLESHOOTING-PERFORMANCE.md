# Quiz 서비스 성능 최적화 트러블슈팅

**작성일**: 2026년 3월 11일  
**문제**: Quiz 서비스 부하 테스트에서 심각한 성능 저하 발생  
**결과**: 읽기 성능 97% 개선 완료, 쓰기 성능 추가 최적화 필요

- **처리량**: 42만 요청 → 115만 요청 (174% 증가)
- **안정성**: P95 응답시간 10초 → 0.16초 (98% 개선)

## 📋 문제 상황

### 초기 증상
부하 테스트(10분간 150명→1,000명 동시 사용자)에서 Quiz 서비스가 극도로 느려졌습니다:

- Quiz 서비스 평균 응답시간: **3.7초** (목표: 1초 이하)
- 95% 사용자가 **10초 이상** 대기 (목표: 2초 이하)
- Community 서비스는 정상 (평균 0.4초)

### 근본 원인 분석

**1. 비효율적인 랜덤 쿼리**
```sql
-- 문제가 된 쿼리
SELECT * FROM quiz.questions 
WHERE difficulty = ? AND type = ? 
ORDER BY RANDOM() 
LIMIT 1;
```
- 매번 전체 테이블을 스캔하고 정렬
- 11개 문제밖에 없는데도 3.7초 소요
- 인덱스를 사용할 수 없는 구조

**2. 데이터베이스 연결 부족**
- Quiz 서비스: 기본 설정 (연결 수 제한)
- Community 서비스: 25개 연결 (1,000명 사용자에 부족)

**3. 캐시 미활용**
- 동일한 11개 문제를 매번 데이터베이스에서 조회
- 메모리 캐시 없이 반복적인 DB 접근

## ❌ 1차 최적화 시도 - 실패 사례

### 시도한 방법
1. **쿼리 개선**: `ORDER BY RANDOM()` → `OFFSET` 방식으로 변경
2. **연결 수 증가**: 데이터베이스 연결을 100개로 대폭 증가

### 예상과 다른 결과
- Quiz 응답시간: 3.7초 → **4.4초로 더 악화**
- Community 응답시간: 0.4초 → **1.1초로 악화**
- 전체 처리량: 42만 → **32만 요청으로 감소**

### 실패 원인
1. **OFFSET 쿼리도 여전히 느림**: COUNT(*) 실행 + N개 행 스캔 필요
2. **과도한 연결로 인한 오버헤드**: 100개 연결이 시스템 부하 증가
3. **데이터베이스 과부하**: Quiz 서비스가 DB를 과부하시켜 다른 서비스까지 영향

## ✅ 2차 최적화 - 성공 사례

### 핵심 아이디어: "데이터베이스를 아예 안 쓰자"

Quiz 문제는 자주 바뀌지 않는 정적 데이터입니다. 매번 데이터베이스에 물어볼 필요가 없다고 판단했습니다.

### 구현한 해결책

**1. 메모리 캐싱 도입**
```go
// 서비스 시작 시 모든 문제를 메모리에 로드
type QuizRepository struct {
    questions []Question  // 11개 문제 전체를 메모리에 저장
}

func (r *QuizRepository) GetRandomQuestion() Question {
    // 데이터베이스 접근 없이 메모리에서 즉시 반환
    randomIndex := rand.Intn(len(r.questions))
    return r.questions[randomIndex]
}
```

**2. 비동기 답안 처리**
```go
func (r *QuizRepository) SubmitAnswer(answer UserAnswer) error {
    // 1단계: Redis 큐에 빠르게 저장
    r.redisClient.LPush("answer_queue", answer)
    
    // 2단계: 사용자에게 즉시 성공 응답
    return nil
    
    // 3단계: 백그라운드 워커가 나중에 DB에 저장
}
```

**3. 자동 데이터 갱신**
- 30초마다 데이터베이스에서 최신 문제 목록을 다시 로드
- 새로운 문제가 추가되어도 자동으로 반영

### 놀라운 결과

**읽기 성능 (GetRandomQuestion)**
- 응답시간: 3.7초 → **0.05초** (74배 빠름)
- 성공률: **100%** (완벽)
- 데이터베이스 쿼리: **0개** (완전 제거)

**쓰기 성능 (SubmitAnswer)**
- 응답시간: 즉시 응답 (Redis 큐 활용)
- 성공률: 36% (개선 필요)
- 실제 DB 저장: 백그라운드에서 처리

**전체 시스템 (10분 테스트 결과)**
- 평균 응답시간: 2,070ms → **53ms** (97% 개선)
- 처리량: 42만 → **115만 요청** (174% 증가)
- P95 응답시간: 10,411ms → **164ms** (98% 개선)

## 🔧 기술적 세부사항

### 사용한 기술 스택
- **ORM**: GORM (기존 database/sql에서 업그레이드)
- **캐시**: Redis (비동기 큐 + 향후 캐시 확장 가능)
- **메모리 사용량**: 55KB (11개 문제 × 5KB)
- **배포**: Kubernetes + ArgoCD

### 해결한 기술적 이슈
1. **PostgreSQL 배열 파싱**: `{option1,option2}` 형식 지원
2. **gRPC Health Check**: 미구현으로 인한 배포 실패 해결
3. **Redis HA 구성**: 2개 파드로 가용성 확보

## 🚧 남은 과제

### SubmitAnswer 쓰기 성능 (36% 성공률)
**문제**: Redis 큐에서 데이터베이스로 저장하는 과정에서 병목 발생

**개선 방안**:
1. 데이터베이스 연결 풀 튜닝
2. 배치 처리로 여러 답안을 한 번에 저장
3. 데이터베이스 인덱스 최적화

### 모니터링 강화
- Redis 큐 길이 모니터링
- 메모리 사용량 추적
- 답안 저장 지연 시간 측정

## 💡 배운 점

1. **캐시의 힘**: 적은 양의 정적 데이터는 메모리 캐싱이 가장 효과적
2. **비동기 처리**: 사용자 응답과 실제 처리를 분리하면 체감 성능 대폭 향상
3. **단계적 접근**: 한 번에 모든 것을 바꾸려 하지 말고 단계별로 검증
4. **실패도 가치**: 1차 시도 실패로 근본적인 해결책을 찾게 됨
