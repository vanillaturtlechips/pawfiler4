# 2026-03-19 리포트 인프라 배포 및 데이터 불일치 이슈

## 1. Lambda Function URL → API Gateway 전환

### 문제

리포트 생성 Lambda에 Function URL을 사용했는데 403 에러 발생.

### 원인

AWS Organizations SCP(Service Control Policy)에서 `lambda:InvokeFunctionUrl` 액션이 차단되어 있었음.
`lambda:InvokeFunction`은 허용되어 있어서 API Gateway를 통한 호출은 정상 동작.

### 해결

Function URL을 제거하고 API Gateway HTTP API로 대체.

**변경된 리소스 (terraform/modules/lambda_report/main.tf)**

- `aws_apigatewayv2_api` — `pawfiler-report-api`
- `aws_apigatewayv2_integration` + `aws_apigatewayv2_route` (POST/GET /generate)
- `aws_apigatewayv2_stage` — `$default`
- `aws_lambda_permission` — `AllowAPIGatewayInvoke`

**API Gateway URL**

```
https://gg7jhcxbsc.execute-api.ap-northeast-2.amazonaws.com
```

**GitHub Secret 업데이트**

- `VITE_REPORT_BASE_URL` → 위 URL로 설정
- 프론트엔드 재배포 후 적용됨

## 2. 리포트 데이터 불일치 (XP/코인/횟수)

### 증상

퀴즈를 풀고 리포트 생성 시 프로필 페이지와 수치가 다르게 표시됨.

- 총 횟수/정답률: 배치 처리 지연으로 직후 불일치
- XP/코인: `AddRewards` gRPC 비동기 처리 지연으로 불일치

### 원인

**총 횟수/정답률**

`quiz/internal/repository/gorm_repository.go`의 `SaveAnswer` 흐름:

```
답변 제출
  → Redis queue에 즉시 저장 (user_stats는 바로 업데이트)
  → 10개 쌓이거나 배치 워커 주기 도달 시 user_answers DB에 flush
```

리포트가 `user_answers`를 직접 COUNT하면 배치 처리 전 데이터 누락.

**XP/코인**

`AddRewards` gRPC가 `go func()` 비동기로 실행되어 `quiz.user_profiles` 업데이트 전에 리포트가 DB를 읽으면 이전 값이 조회됨.

### 해결 (2026-03-23 적용)

**총 횟수/정답률** — `fetch_user_report_data`에서 `user_answers COUNT` 대신 `user_stats` 테이블 직접 조회로 변경.

```sql
-- 변경 전: user_answers COUNT (배치 지연 있음)
-- 변경 후:
SELECT us.total_answered as total,
       us.correct_count as correct,
       up.total_exp as total_xp
FROM quiz.user_stats us
JOIN quiz.user_profiles up ON us.user_id = up.user_id
WHERE us.user_id = %s
```

**XP** — `build_html`에서 DB 조회값(`stats['total_xp']`) 대신 프론트에서 전달한 `total_exp` 파라미터 직접 사용.

```python
# 변경 전
xp = int(stats['total_xp'] or 0)
# 변경 후
xp = total_exp  # 프론트 quizProfile.totalExp (최신값)
```

**코인/티어** — `ReportRequest`에 `total_exp`, `total_coins`, `tier_name` 필드 추가, 프론트 `quizProfile` 값 우선 사용.

**수정 파일**

- `backend/services/report/main.py` — `fetch_user_report_data`, `build_html`, `ReportRequest`
- `frontend/src/lib/api.ts` — `generateReport`에 `quizProfile` 파라미터 추가
- `frontend/src/pages/ProfilePage.tsx` — 리포트 생성 시 현재 `quizProfile` 전달

## 3. Lambda 이미지 경량화 (옵션 B, 2026-03-23 적용)

### 배경

Lambda 이미지 465MB → cold start 3-4초 문제.

### 분석

| 항목            | 크기   | 제거 여부                                        |
| --------------- | ------ | ------------------------------------------------ |
| scipy           | ~130MB | ✅ `make_parchment_bg`(미사용 함수)에서만 사용   |
| Pillow          | ~15MB  | ✅ import조차 없음                               |
| uvicorn         | ~3MB   | ✅ Lambda+mangum 조합에서 불필요                 |
| NotoSansCJK OTC | ~50MB  | ✅ KR OTF 서브셋으로 교체 (~4MB)                 |
| gcc, gcc-c++    | ~50MB  | ❌ numpy/contourpy meson 빌드에 필수 — 제거 불가 |

### 변경 내용

**requirements.txt** — scipy, Pillow, uvicorn 제거. matplotlib 3.8.4 → 3.9.4 업그레이드

**Dockerfile**

- `gcc gcc-c++` 유지 필수 — numpy/matplotlib의 `contourpy` 의존성이 meson 기반 C++ 빌드를 요구함. 제거 시 CI 빌드 실패
- 폰트: `NotoSansCJK.ttc` (OTC 전체, ~50MB) → `NotoSansCJKkr-Regular.otf` (KR 서브셋, ~4MB)
- CI `build-push-action`에 `platforms: linux/amd64` 명시 추가

**main.py**

- `from scipy.ndimage import gaussian_filter` import 제거
- `make_parchment_bg` 함수 삭제 (호출되지 않는 dead code)
- 중복 정의된 첫 번째 `get_grade` (발자국 그리는 dead code) 삭제
- 폰트 경로 `NotoSansCJK.ttc` → `NotoSansCJKkr-Regular.otf` 업데이트

### CI 빌드 시 주의사항

`gcc gcc-c++` 제거 불가. numpy/contourpy 최신 버전은 meson 빌드를 시도하며 C++ 컴파일러를 요구함. pre-built wheel이 없는 버전이 pip에 의해 선택될 수 있으므로 gcc는 반드시 유지.

### 결과

실제 이미지 크기: ~271MB (465MB → 42% 감소)
cold start: 3-4초 → 약 2초 개선 (scipy 제거 효과)
